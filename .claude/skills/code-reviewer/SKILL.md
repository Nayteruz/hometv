Когда пользователь вызывает `/code-reviewer`, выполни полный аудит кода проекта. По умолчанию проверяются все изменённые файлы (рабочая копия git). Если аргумент `--strict` — проверяется ВЕСЬ проект. Если передан путь/файл — проверяется только указанная область.

## Процесс ревью

### Шаг 1: Определи область проверки

```bash
# Если --strict или нет изменений — проверяем весь src/
git diff --name-only          # изменённые файлы
git diff --cached --name-only # staged файлы
```

- **По умолчанию**: проверяем изменённые файлы (modified + staged)
- **`--strict`**: проверяем ВСЕ файлы в `src/`
- **Указан путь**: проверяем только этот файл/директорию

### Шаг 2: Проверь каждый файл по 6 категориям

Для каждого проверяемого файла пройдись по всем категориям ниже. Каждая находка получает уровень критичности:

- 🔴 **CRITICAL** — ошибка, которая сломает приложение, вызовет утечку данных или нарушит безопасность. Нужно исправить немедленно.
- 🟠 **WARNING** — нарушение стандартов проекта, потенциальный баг, плохая практика. Нужно исправить.
- 🟡 **SUGGESTION** — код работает, но можно улучшить (производительность, читаемость, переиспользование).

#### Категория 1: Типизация TypeScript

Проверяй каждый файл на соответствие строгой типизации:

- **CRITICAL**: Использование `any` без необходимости (особенно в сигнатурах функций, пропсах, стейте)
- **WARNING**: Неявный `any` в параметрах функций или возвращаемых значениях
- **WARNING**: Отсутствие интерфейса/типа для сложных объектов
- **WARNING**: Использование `as` (type assertion) без крайней необходимости — предпочитай type guards
- **SUGGESTION**: Можно использовать `satisfies` вместо аннотации типа (TS 4.9+)
- **SUGGESTION**: Избыточная аннотация типа, где TypeScript выводит тип сам
- **SUGGESTION**: Для `defineProps` используй дженерик `defineProps<T>()` вместо `as` каста

Правила проекта:
- Все пропсы через `defineProps<T>()` + `withDefaults()`
- Все emits через `defineEmits<{...}>()`
- Импорт типов через `import type { ... }`
- Интерфейсы с префиксом `I` (например `IFilmEntity`)
- Сторы типизированы явно `(): IStoreState => ({...})`

#### Категория 2: Безопасность

- **CRITICAL**: API ключи, токены или секреты в клиентском коде (`.vue`, `.ts`, `.js` файлы в `src/`)
- **CRITICAL**: `innerHTML` / `v-html` с пользовательскими данными — XSS уязвимость
- **CRITICAL**: Отсутствие валидации пользовательского ввода перед отправкой в Firestore
- **CRITICAL**: `dangerouslyDisableSandbox: true` или другие обходы безопасности
- **WARNING**: Использование `eval()`, `new Function()`, `setTimeout(string)`
- **WARNING**: Непроверенные редиректы (`window.location.href` с параметрами из URL)
- **WARNING**: Отсутствие проверки прав доступа перед операциями с Firestore
- **SUGGESTION**: Отсутствие санитизации вывода (например, названия фильмов могут содержать HTML)

Специфика проекта:
- Firebase config в `firebaseConfig.ts` — проверь, что это публичные ключи (не секретные)
- API ключи пользователей хранятся в Firestore — проверь, что они не логируются и не видны в DOM
- Все операции с Firestore должны проходить проверку `if (!this.user)` перед выполнением

#### Категория 3: Vue компоненты и разметка

- **CRITICAL**: Мутация пропсов напрямую (props are read-only)
- **WARNING**: Отсутствие `<style scoped>` в компоненте (или `<style scoped lang="scss">`)
- **WARNING**: Использование Options API вместо `<script setup>` (нарушение конвенции проекта)
- **WARNING**: `v-for` без `:key` с уникальным значением
- **WARNING**: Большие вычисления в шаблоне вместо `computed`
- **WARNING**: Мутация состояния Pinia напрямую в компоненте (должно быть через actions)
- **SUGGESTION**: Компонент можно разбить на более мелкие (слишком большой — >300 строк)
- **SUGGESTION**: `v-if` + `v-for` на одном элементе (предпочитай `computed` с фильтрацией)
- **SUGGESTION**: Отсутствие `KeepAlive` для компонентов, которые дорого перерендеривать

Правила проекта:
- Всегда `<script setup lang="ts">`
- Всегда `<style scoped lang="scss">`
- Импорты через `@/` (не относительные пути)
- Для кросскомпонентной коммуникации: `inject('emitter')` из `mitt`, НЕ props drilling

#### Категория 4: Хранилища (Pinia Stores)

- **WARNING**: Прямая мутация состояния вне экшена (например `store.films.push(...)` в компоненте)
- **WARNING**: Асинхронные операции без обработки ошибок (try/catch)
- **WARNING**: Отсутствие проверки `this.user` перед операциями с Firestore
- **SUGGESTION**: Дублирование логики между сторами — можно вынести в `utils.ts`
- **SUGGESTION**: getter можно заменить на простую функцию (нет реактивности/кеширования)

Правила проекта:
- Options API стиль: `defineStore('name', { state, getters, actions })`
- State factory возвращает типизированный объект
- Все мутации списков через `safeUpdateUserData(fieldName, updaterFn)`
- После мутации обновлять локальный state немедленно

#### Категория 5: API и данные

- **CRITICAL**: Отсутствие обработки ошибок API (сетевые ошибки, таймауты, 4xx/5xx)
- **WARNING**: Не используется `AbortSignal` для отменяемых запросов
- **WARNING**: Данные фильмов не нормализованы через `getFilmEntity()` / `getFilmEntityList()`
- **WARNING**: Жёстко закодированные URL вместо использования констант из `const.ts`
- **SUGGESTION**: Отсутствие дебаунса/троттла для поисковых запросов
- **SUGGESTION**: Дублирование логики запросов — можно вынести в `api.ts`

Правила проекта:
- Все API-вызовы через `src/components/api.ts`
- Данные фильмов нормализуются через `getFilmEntity()` (поля `kinopoiskId`/`filmId`/`id`, `posterUrlPreview`/`posterUrl`/`image`)
- Ошибки API: ловить `ApiError` (содержит `status`, `url`, `message`)
- Ошибки Firebase: ловить `FirebaseError`, использовать `translateErrorCode()`

#### Категория 6: Общее качество кода

- **WARNING**: Неиспользуемый код (импорты, переменные, функции) — `noUnusedLocals: true` в tsconfig
- **WARNING**: `console.log` в продакшн-коде (кроме `console.error`/`console.warn`)
- **SUGGESTION**: Нарушение SOLID (слишком много ответственностей у одного модуля)
- **SUGGESTION**: Нарушение KISS (излишняя сложность, паттерны ради паттернов)
- **SUGGESTION**: Магические числа/строки вместо именованных констант
- **SUGGESTION**: Длинная функция (>50 строк) — можно разбить
- **SUGGESTION**: Слишком много параметров у функции (>4)
- **SUGGESTION**: Комментарий объясняет «что», а не «почему» (код должен быть самодокументирован)
- **SUGGESTION**: Неинформативные имена переменных (`data`, `item`, `val`, `tmp`)

## Важные замечания

- **Не проверяй** то, что отлавливается линтером или type-check'ом — форматирование, неиспользуемые переменные (если только это не указывает на баг), стиль отступов. Предполагай, что `npm run lint` и `npm run type-check` выполняются отдельно.
- **Не считай проблемами** код, который уже закомментирован как `// eslint-disable` или `// @ts-ignore` с пояснением.
- **Учитывай контекст**: если файл — демо-данные (как `favorites.js`), не требуй от него строгой типизации.
- **Ссылайся на CLAUDE.md** когда находишь нарушение: «Нарушение CLAUDE.md: *цитата правила*».
- **Ссылайся на конкретные строки** в формате `путь/к/файлу.ts:номер`.
- Если область проверки большая (>15 файлов), используй параллельных агентов для ускорения: по одному на каждую категорию.
- Фокусируйся на том, что действительно важно. Лучше 5 точных находок, чем 50 ложных срабатываний.
